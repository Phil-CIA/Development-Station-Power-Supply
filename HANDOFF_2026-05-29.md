## UART Bug — Updated Root Cause Analysis

### What was confirmed via Elecrow wiki (May 29 research)

CrowPanel Advance 4.3" V1.1 UART1-OUT connector (HY2.0-4P, labeled J?) pinout:
- pin 1 = GND
- pin 2 = 3V3
- pin 3 = RX (IO19)
- pin 4 = TX (IO20)

**I2S MIC shares the same pins:**
- MIC_SD = IO20
- MIC_CLK = IO19

**Function-select switch table for V1.1:**

| SW1 | SW2 | Active function |
|-----|-----|-----------------|
| 0   | 0   | MIC & SPK       |
| 0   | 1   | WM (wireless module) |
| 1   | 1   | MIC & TF Card   |

> Note: V1.0 had a `1,0` state; V1.1 replaced it with `1,1 = MIC & TF Card`.

**Key finding:** In no switch position are IO19/IO20 disconnected from the MIC chip hardware traces. The I2S MIC chip output drivers appear to be permanently attached to IO19/IO20 at the PCB level regardless of the function-select state.

### What has been tried

- Clean UART signal confirmed at JST UART1-OUT pads (Kingst LA1010) ✅
- Physical wire TX↔RX swap at HAT J8: no change ❌
- Firmware TX/RX pin swap: no change ❌
- All three switch positions tested: no change ❌
- PROBE command on CrowPanel IO19: ~4M HIGH / ≤3 LOW / ≤6 transitions over 7 s — signal not arriving

### What has NOT been done yet

- **Ohm test from JST UART1-OUT pads to S3 IO19/IO20 pads** (rules out PCB trace break vs. active mux)
- Routing Serial1 to alternate S3 GPIOs (bypasses IO19/IO20 entirely)

## Next Session: UART Debug Plan

### Step 1 — Ohm the PCB trace (bench work, ~5 min)

With board **unpowered**, measure continuity:
- JST UART1-OUT pin 3 → S3 module pad IO19
- JST UART1-OUT pin 4 → S3 module pad IO20

**Expected outcomes:**
- Continuity present → trace is intact; the I2S MIC chip or a mux IC is blocking the signal electrically
- Open → trace break or wrong pin assumed → respin assumption, recheck connector orientation

### Step 2 — Re-add PROBE command to CrowPanel firmware

Add back the IO pin probe diagnostic to `crowpanel-43-bringup/src/main.cpp`:
```cpp
// In Serial command handler:
if (cmd.startsWith("PROBE")) {
  uint32_t ms = cmd.length() > 5 ? cmd.substring(6).toInt() : 6000;
  if (ms == 0) ms = 6000;
  Serial.printf("PROBE IO19 for %u ms...\n", ms);
  Serial1.end();
  gpio_reset_pin((gpio_num_t)19);
  pinMode(19, INPUT);
  uint32_t hi=0, lo=0, trans=0;
  bool last = digitalRead(19);
  uint32_t t0 = millis();
  while (millis()-t0 < ms) {
    bool v = digitalRead(19);
    if (v != last) { trans++; last=v; }
    if (v) hi++; else lo++;
  }
  Serial.printf("PROBE done: hi=%u lo=%u trans=%u\n", hi, lo, trans);
}
```

### Step 3 — Try alternate S3 GPIO pair

If trace is intact (IO19/IO20 blocked by MIC chip driver), remap Serial1 to free GPIOs:

Candidate free pins on S3 (check against CrowPanel Advance 4.3" expansion header before assuming free):
- **IO4 / IO5** — test first
- **IO35 / IO36** — check if used by SD or other peripheral

Firmware change: in `disp_link_slave.cpp`, change `Serial1.begin(115200, SERIAL_8N1, RX, TX)` pin constants. Run PROBE on the new RX pin. If signal seen → confirms IO19/IO20 blocked; wire to the new pins.

### Step 4 — Check I2S MIC chip (if trace is intact)

Look at the I2S MIC chip (ICS-43434 or similar) datasheet to confirm:
- Does it actively drive MIC_SD/MIC_CLK lines when not selected?
- Is there a chip-enable or power-down input that could be driven low?

If so, add GPIO control of the MIC chip power-down/enable pin from firmware before asserting Serial1.
